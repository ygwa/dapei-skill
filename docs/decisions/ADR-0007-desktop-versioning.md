## Problem Statement

`dapei-skill` ships as a single Skill invoked via `@dapei ...`. The new `desktop/` subtree is a separate Electron application that depends on the same engine (`engine/dapei-engine.ts`) and `@dapei/core`, but targets a different delivery surface (a local desktop binary, not a skill). Two questions need an answer before any desktop PRs can be cut:

1. **Release cadence**: does the desktop ship on every engine release, or only when it has user-visible changes?
2. **Version surface**: does the desktop carry its own semver (`@dapei/desktop-app` at `0.1.0`), share the engine semver, or both?

The `desktop/package.json` was scaffolded at `0.1.0` (private) and the root `package.json` is at `3.2.0`. The two are unrelated numbers. The 13-milestone plan in `.omo/plans/desktop-m1-m2.md` adds real work that mutates both `desktop/` and `packages/core/`. We need a versioning rule that does not force every desktop change to bump the engine version, and does not let a desktop change slip into a "skill" release by accident.

## Constraints

- `desktop/` is a **nested pnpm workspace** with its own `pnpm-workspace.yaml` and `pnpm-lock.yaml`. The root `pnpm-workspace.yaml` does **not** include `desktop/`.
- `desktop/apps/electron` imports `@dapei/desktop-*` packages (workspace) and at runtime spawns `engine/dapei-engine.ts` via `node --experimental-strip-types`. The desktop has no npm dependency on `@dapei/core` directly — it speaks to the engine through the subprocess boundary.
- Engine capability contract is the public API surface. A desktop that requires a non-existent capability must add it (per ADR-0003 evidence discipline and the "missing → append" rule from `.omo/plans/desktop-m1-m2.md` §0). Desktop-only changes must not require a Skill release.
- The existing `scripts/lib/release-version.mjs` keeps the engine, skill, and 6 version sources in sync. Adding the desktop to that machinery mid-M1 would be premature.

## Decision

The desktop follows **"co-major, independent minor, internal canary tag"**:

| Surface | Source of truth | Bump rule |
|---|---|---|
| `desktop/package.json` (`@dapei/desktop-app`, `@dapei/desktop-*`) | independent minor | each milestone that ships user-visible change |
| `desktop/pnpm-lock.yaml` | regenerated by `pnpm install` on desktop changes | deterministic, no manual edit |
| Root `package.json` (skill / engine) | unchanged by desktop work | only bumps when desktop work adds engine capability or fix |
| Root `CHANGELOG.md` | adds a "## Desktop" section per desktop milestone | authored in the same PR that bumps `desktop/package.json` |

Concretely:

- The first desktop PR that lands (M1-0) **bump `desktop/package.json` from `0.1.0` to `0.1.0-canary.0`** and adds a `## Desktop (M0 scaffolded)` CHANGELOG section. This is internal; not a release.
- When M1 ships (after M1-7's PR is reviewed and merged), bump desktop to `0.2.0` and add `## Desktop (M1)` CHANGELOG. The root `package.json` is bumped only if the PR adds engine capability or fixes a shared bug; otherwise it stays.
- When M2 ships, `0.3.0`. The `desktop/package.json` minor is the public marker for "desktop can do new things".
- The desktop **never** publishes to npm during M1+M2 (per the user's decision recorded in `.omo/plans/desktop-m1-m2.md` §0 — "暂不发版, 只仓库内 dev"). `pnpm install` is the only distribution channel during this period.
- Desktop-only commits do not require a root version bump. Skill/engine commits may include desktop changes as a separate commit in the same PR, but the commit messages must separate the two concerns.

## Alternatives Considered

### Option A: Single version, same bump
- **Pros:** Simpler — `desktop/package.json` always matches root `package.json`.
- **Cons:** Forces every desktop-only milestone to bump the Skill version, polluting the Skill release notes. Locks desktop and engine into the same release train even when there is no cross-cutting change.

### Option B: Independent version, no synchronization
- **Pros:** Maximum independence. Desktop can ship daily.
- **Cons:** Risk of drift: a desktop that requires a not-yet-shipped engine capability will be installable from a `desktop/pnpm-lock.yaml` that pins the older engine commit, and the user sees a confusing "engine returned UNKNOWN_CAPABILITY" error. We have to add a runtime check (`engine.run({capabilityId: 'workspace.status'})` + retry with stub) anyway, so the version surface doesn't add safety.

### Option C: Co-major, independent minor, internal canary tag (chosen)
- **Pros:** Clean separation. CHANGELOG stays the audit log. The engine contract (capability IDs + Request/Response shape) is the only true dependency; the lockfile + capability-version handshake already catch drift. Can later graduate to npm publish without re-shaping the version story.
- **Cons:** Two `package.json` versions to keep in mind when reading PRs. The 6-version-source discipline from ADR-0001 must be extended to include `desktop/package.json` when the desktop ships externally.

## Consequences

### Positive
- Desktop milestones are reviewable and shippable on their own; no need to time them with skill releases.
- The CHANGELOG "## Desktop" section gives the user a single document to scan for "what changed in the desktop app".
- The 0.1.0-canary.0 marker is honest: the desktop is real code, but the user explicitly chose to not publish yet.
- The version rule is reversible — graduating to npm publish later is a 30-line change to `scripts/lib/release-version.mjs` + adding a `release:desktop` script, not a refactor.

### Negative
- The desktop does not enter `scripts/lib/release-version.mjs` yet. A future maintainer might forget to bump it on a desktop change. Mitigation: every desktop PR must run `pnpm --filter @dapei/desktop-app typecheck` + the desktop CI job (when added) checks that `CHANGELOG.md` mentions the same milestone marker as the bump.
- The "co-major" rule means a breaking change to the desktop's IPC contract (e.g., renaming `dapei:workspace:open` to `dapei:workspace:load`) would force a major version bump on the desktop, and at that point we'd have to coordinate with the engine's major too. This is acceptable — breaking the IPC contract *should* be a major.

### Neutral
- The 6-version-source discipline from ADR-0001 is unchanged for now. We add a 7th source (`desktop/package.json`) only when desktop ships externally.
