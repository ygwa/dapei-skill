## Problem Statement

PR-4a (Workbench bug fixes) needs to pair `tool:call` with the
matching `tool:result` reliably. The current renderer logic
(`FeatureWorkbenchView.tsx:82-83`) matches by **tool name**, which
breaks when the same tool is called twice in quick succession
(`workspace.status` × 2 — the second `tool:result` updates the
first `tool:call` card, not the second).

The fix requires a unique id on the call side that the result side
echoes back. The id is generated by whoever emits `tool:call` —
the agent backend (mock or opencode).

## Constraints

- Backward compatible — the renderer must keep working when only
  one of the two events carries `toolCallId`. Match by id when
  present; fall back to name match when not.
- Mock agent backend (`packages/agent/src/backends/mock-backend.ts:37-38`)
  must populate the id so renderer can pair correctly.
- Real opencode backend (`packages/agent/src/backends/opencode-backend.ts`)
  must also populate the id; if opencode doesn't provide one,
  generate a uuid in the wrapper.
- Pre-M3-UI tools that don't yet carry the id still pair by name.

## Decision

Extend `AgentEvent` union with `toolCallId?: string` on the two
affected variants:

```ts
// packages/contracts/src/events/agent.ts
export type AgentEvent =
  | { type: "session:ready"; sessionId: string }
  | { type: "session:closed"; sessionId: string }
  | { type: "message:user"; text: string }
  | { type: "message:assistant"; text: string; stage?: string }
  | { type: "tool:call"; name: string; input: unknown; toolCallId?: string }
  | { type: "tool:result"; name: string; output: unknown; ok: boolean; toolCallId?: string }
  | { type: "capability:invoked"; id: string; ok: boolean };
```

The id is `crypto.randomUUID()` at emission time. Mock agent
generates one uuid per scripted `tool:call`; the matching
`tool:result` reuses the same id.

### Renderer pairing logic (PR-4a)

```ts
// pseudo-code for FeatureWorkbenchView's effect
case "tool:result": {
  const id = e.toolCallId;
  setMessages(prev => prev.map(m => {
    if (m.kind !== "tool" || m.meta?.toolName !== e.name) return m;
    // Prefer id match; fall back to "latest matching tool"
    const isIdMatch = id && m.meta?.toolCallId === id;
    const isLatest = !id && m.meta?.pending;
    return (isIdMatch || isLatest) ? { ...m, meta: {...m.meta, toolOk: e.ok, ...} } : m;
  }));
}
```

This is forward-compatible: when id is missing (legacy or non-id
backends), the renderer falls back to "latest matching tool" —
same behavior as today, just with the old buggy assumption
explicitly named.

## Consequences

### Positive

- Duplicate tool calls pair correctly (the headline bug).
- Future opencode backends can opt into ids by passing them through
  the ACP transport (when opencode provides them).

### Negative

- One more field in the AgentEvent union. Test count goes up by
  maybe 2 (mock test asserts id is propagated).

## Verification

- New test in `apps/electron/src/main/agent/__tests__/mock-backend.test.ts`:

- Existing 6 agent tests still pass.

## Open Questions

  should map it to `toolCallId` rather than generating a uuid.
